## Context

Presence v0 (shipped, v0.2.0) renders the **glyph** for a `PresenceMessage` on the 8Ã—8 via
the MCP server's canned-glyph frame push; the board only *stores* the message, and the rich
`data` (progress / readouts / sparkline) shows only on the desktop card. v0.5 teaches the
**board** to render the `data` itself on the LEDs, using the existing `fonts.ino` pixel fonts
and column fills. This is the next step toward the board being a true capability-aware
renderer (the "presence protocol" thesis).

## Decisions (from brainstorming)

1. **Hybrid render split.** The board renders the **data shapes** natively; glyph-only
   (no-`data`) presences keep using the MCP canned-glyph frame push exactly as today. We do
   NOT port the canned glyph art to firmware.
2. **Data owns the panel; color = mood.** A data presence draws the number/bar/sparkline
   across the full 8Ã—8 in the **intent's color** â€” no competing glyph. Mood reads from color
   (matches the low-brightness "color carries emotion" lesson).
3. **`values` cycles.** With 1â€“3 readouts, the LEDs show each value in turn as one big number
   (~1.8 s apart) â€” legibility-first. (Not stacked small numbers.)
4. **`progress` is a bottom-up panel fill** (battery/glass style), `round(p Ã— 64)` pixels lit
   from the bottom up â€” far finer than an 8-step bar.
5. **Transient.** Presence-data mode is not NVS-auto-resumed (like text/sketch), so a reboot
   comes up on the last real animation, not a stale empty data screen.
6. **Ships as a new minor version** (0.2.0 â†’ 0.3.0): touches firmware AND MCP.

## Architecture

### New board render mode: `animationName == "presence"`
- New file `esp32_matrix_webserver/anim_presence.ino` with `parsePresence()` + `runPresenceFrame()`.
- **Parse once, not per frame** (heap discipline â€” this project has a documented heap-crash
  failure mode): when presence mode is entered, `parsePresence()` parses the stored
  `presenceJson` (ArduinoJson v7 `JsonDocument`) ONCE into render-ready globals â€” the intent
  `CRGB`, a shape enum (`PRES_NONE | PRES_PROGRESS | PRES_VALUES | PRES_SERIES`), a `float
  progress`, a small `float values[3]` + count, and a `uint8_t series[8]` of pre-normalized
  column heights + count. `runPresenceFrame()` then only *draws* from those globals each frame
  (no JSON touched in the render loop). `applyAnimationBody` calls `parsePresence()` when it
  sets `animationName = "presence"`.
- Dispatch branch added to the main loop: `else if (animationName == "presence") runPresenceFrame();`
- `applyAnimationBody` accepts `type == "presence"`, calls `parsePresence()`, sets
  `animationName = "presence"`, and the mode is excluded from NVS auto-resume (transient, like `text`).

### Board intentâ†’color table
- A small firmware helper `CRGB presenceColor(const String& intent)` mirroring the card's
  `PRESENCE_VOCAB` colors so the 8Ã—8 and card agree: working/alert = amber, thinking/question
  = blue, done/ok = green, celebrate = magenta, error = red, info = slate, idle = dim indigo.
  Unknown intent â†’ neutral white. (Lives in `anim_presence.ino`.) Note `COLOR_ORDER` is `RGB`,
  so `CRGB(0xRR,0xGG,0xBB)` maps straight through.

### `presence_set` routing (MCP, `index.ts`)
`presence_set` always `POST`s `/api/presence` (the card always renders the full message). Then:
- **`data` present** â†’ `POST /api/display/animation { type: "presence" }` â†’ the board renders
  the data natively.
- **no `data`** â†’ render the canned glyph via the existing `CANNED` â†’ `expressionToWire` â†’
  `/api/display/frames` path (today's behavior).

So one tool call drives: card (always, rich) + 8Ã—8 (data natively when present, glyph
otherwise). `normalizePresence` already returns `data` only when present, so the branch is
`if (msg.data) â€¦`.

## Per-shape rendering (data fills 8Ã—8, drawn in `presenceColor(intent)`)

### `progress` (0..1) â†’ bottom-up panel fill
`n = round(clamp(p,0,1) Ã— 64)` pixels lit from the bottom. Fill whole rows from the bottom row
(y=7) upward; the partially-filled top row lights its leftmost `n % 8` columns. Battery/glass
feel; ~1.5% precision.

### `series` â†’ column sparkline
Take the last `min(8, len)` points. Normalize over their own min..max to a height `1..8`
(flat series â†’ a centered mid-height line, height 4). Draw each point as a column filled from
the bottom up; if fewer than 8 points, center the columns horizontally.

### `values` (1..3 readouts) â†’ cycling big number
Render `values[i].value` rounded to an integer as a 3Ã—5 number, vertically centered. Cycle
`i` through the array every ~1.8 s (module-level index + `millis()` timer). **No unit/label on
the LEDs** â€” the card carries units/labels. Number sizing:
- â‰¤ 2 chars (covers âˆ’9â€¦99) â†’ drawn static, horizontally centered (`fontCenterCol`).
- Longer (e.g. 3+ digits, or a leading `-` plus 2 digits) â†’ horizontally scrolled 3Ã—5 within
  the presence renderer (self-contained scroll offset advanced by `millis()`), so big/negative
  numbers still read.

## Error handling

- **`animationName=="presence"` but the stored message has no `data`** (a no-data message, or
  post-reboot idle default) â†’ render nothing / a neutral dim state; never garbage-draw.
- **Malformed/oversized `presenceJson`** is already guarded by the v0 `/api/presence` POST
  handler (body cap + heap guard + intent check); on top of that `parsePresence()` sets the
  shape to `PRES_NONE` if the parse fails or `data` is absent, and `runPresenceFrame()` no-ops
  on `PRES_NONE`.
- **`progress`** clamped to 0..1. **`series`** with `max==min` â†’ mid-height line (no divide by
  zero). **`values`** non-finite â†’ skip that entry; empty after filtering â†’ neutral.
- **Unknown intent** â†’ neutral white from `presenceColor`.

## Testing

- **MCP routing** (unit/smoke): `presence_set` sends `{type:"presence"}` when `msg.data` is
  present and uses the glyph-frame path when it is absent; both still POST `/api/presence`.
- **Firmware** (cannot compile here â€” hardware-verified): flash, then drive each shape via
  `presence_set` at brightness 5 and confirm â€”
  - `progress` 0.0 / 0.5 / 0.73 / 1.0 fills bottom-up to the right level;
  - `series` renders a column chart whose shape matches the input (rising/spiky);
  - `values` shows the number legibly, cycles through 2â€“3 values, and a 3-digit value scrolls;
  - color matches the intent on each;
  - a no-`data` presence still shows the canned glyph (unchanged);
  - reboot does NOT come back into an empty presence screen (transient).
- Card rendering is unchanged from v0.

## Out of scope (YAGNI â€” beyond v0.5)

- Units/labels on the 8Ã—8 (the card's job).
- Glyph + data simultaneously (data owns the panel by decision).
- Porting the canned glyph art to firmware (hybrid keeps MCP glyphs).
- NVS persistence of presence data across reboots.
- Animated/eased transitions between values (a plain cycle is enough).
- More than 3 values or >8 visible sparkline columns (caps from the v0 IR stand).
