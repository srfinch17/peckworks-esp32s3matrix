PART 01

What it is: a display that's genuinely yours

The whole idea in one line

A tiny standalone panel that runs the show by itself, and that any program can light up.

Most LED-matrix projects are tethered: a sketch on your laptop pushes pixels over USB, and the moment you unplug, it's dark. This firmware flips that. Everything the panel needs lives on the board: the animations, the web UI, the WiFi onboarding, the persistence. Power it over USB-C and it boots straight into whatever it was last showing. Open its address on your phone and you've got a full control panel. Point any HTTP client at it and you're driving pixels from any language.

Plug-in & standalone
Boots into its last animation, brightness, and settings, all persisted to flash. No computer in the loop.
Self-onboarding WiFi
First boot opens a ESP32-Matrix-Setup hotspot; join it, pick your network, done. Then it's at esp32matrix.local.
Its own control panel
A full web UI served from the device: animations, brightness, weather, clock, a pixel painter, a calibration lab.
A real animation library
Fire, matrix rain, fireworks, dance floor, comet, an IMU-reactive liquid, snow, starfield, each tunable live.
Useful, not just pretty
Live weather by ZIP, an NTP clock + countdown timers, scrolling text, and a vibration-reactive VU meter.
An open API
Every capability is an HTTP endpoint: send a built-in effect, push your own frames, set brightness, read sensors.
The shape of it: think of a smart bulb that happens to be an 8×8 canvas: it has an IP address, a web page, and an API, and it keeps glowing whether or not anything is talking to it. Except this one is fully open, runs offline, and you flash it yourself.

PART 03

The HTTP API: the contract anything can drive

One surface, every capability

If it can make an HTTP request, it can drive the panel.

There's no SDK to adopt and no protocol to learn. The board's entire feature set is a small set of JSON endpoints under http://esp32matrix.local. This is also the documented contract that external clients (including the Claude integration) depend on, so it stays stable.

Launch a built-in animation. The transient flag keeps it out of boot-resume:

curl: fire at half intensity
# roar some fire, classic theme
curl -X POST http://esp32matrix.local/api/display/animation \
  -H "Content-Type: application/json" \
  -d '{"type":"fire","transient":true,"intensity":6,"theme":"classic"}'

Push your own animation, up to 24 frames, each 64 RRGGBB pixels in row-major order:

curl: a custom 2-frame blink
# loop:0 = forever
curl -X POST http://esp32matrix.local/api/display/frames \
  -d '{"frames":["…384 hex chars…","…"],"frame_ms":150,"loop":0}'

PART 04

How it actually renders: from a pixel to a lit LED

The part most 8×8 projects get subtly wrong

The grid is plain row-major, not serpentine.

Plenty of LED matrices zig-zag their wiring, so even rows run left-to-right and odd rows run right-to-left. This panel does not. The mapping is the boring, predictable one: XY(x, y) = y * 8 + x, origin top-left, x increasing right, y increasing down. Out-of-bounds returns -1. Getting this assumption right is the difference between a clean image and a mirrored mess on every other row.

The coordinate system. Always draw through setPixel(x, y, CRGB). It's bounds-checked, so an off-grid write is a safe no-op instead of memory you didn't mean to touch.

One more twist: the color order is RGB, not GRB

WS2812B strips almost always expect green-first (GRB). This panel is wired so FastLED's COLOR_ORDER is plain RGB, meaning CRGB(r, g, b) maps straight through. It's a small thing that costs an afternoon if you assume the default.

Calibration without a reflash: the matrixShow() chokepoint

Real LEDs lie. The panel's green runs hot and dim content gets crushed. Rather than bake corrections into every animation, the firmware applies one global per-pixel transform at the single point every frame passes through on its way to the hardware: matrixShow(). It saves the frame, corrects in place, pushes it, then restores the originals, so animation code keeps working on clean leds[] values and never knows calibration happened.

One transform, one place. Calibration is a save → correct-in-place → restore wrapper around the hardware push, so it's always-on, never double-applied, and tunable live from the calibration lab without a firmware rebuild.

PART 05

Drive it with Claude: the panel as an expression channel

What an open HTTP API unlocks

Because the whole board is just an API, it becomes an ambient status light for AI coding sessions.

A separate, public companion project (Claude Expression Studio) turns the panel into Claude's expression channel: a spinner while it works, a checkmark when it's done, a glyph when it's waiting on you. It does this purely by POSTing to the same endpoints documented above. No shared code, no firmware coupling. Just the contract.

  • Renderer-agnostic. The board is one optional renderer; the Studio runs fully board-free too, simulating everything in the browser.
  • A browser studio to design, preview, and wire ~40 animations to moments in a coding session.
  • Couples only over this API. Point it at esp32matrix.local and the same expressions light up real LEDs.

PART 06

Flash one yourself: a ~$15 board and one binary

From zero to a glowing panel

Three steps, no toolchain required.

  1. Get the board.

    The Waveshare ESP32-S3-Matrix, an 8×8 WS2812B grid on a USB-C ESP32-S3. On Amazon ↗

  2. Flash one binary.

    One-click flash in Chrome/Edge, or grab the offline installer (merged .bin + flash.bat/flash.sh). It's a single merged image (app + web UI), so there's nothing to assemble.

  3. Join WiFi.

    Connect to the ESP32-Matrix-Setup hotspot, pick your network, then open esp32matrix.local. From here you're back at Part 02, the onboard control panel.

Want to build from source instead? The firmware is plain Arduino (.ino). See the repo's README for the Arduino IDE setup and library list.
